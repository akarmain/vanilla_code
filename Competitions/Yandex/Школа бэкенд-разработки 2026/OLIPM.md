# OLIPM: стиль олимпиадного кода
## 1) Что видно по твоему коду

- Преобладает практичный стиль: «считать -> посчитать -> вывести», без лишней архитектуры.
- Часто есть `main()` и запуск через `if __name__ == '__main__': main()`.
- Имена переменных обычно короткие: `n`, `m`, `arr`, `ans`, `k`, `s`, `t`. не используй заглавные буквы в именах
- Type hints используются редко.
- `sys.stdin` не используется, ВСЕГДА обычный `input()`.
- Комментарии никогда НЕ пиши

## 2) Базовый шаблон

```python
def main():
    # ввод
    # решение
    # вывод


if __name__ == '__main__':
    main()
```

Если несколько тестов:

```python
def main():
    ...


if __name__ == '__main__':
	for _ in range(int(input())):
    	main()
```

## 3) Как писать, чтобы код выглядел естественно

- Держи решение в 1 файле, без дробления на множество классов/модулей.
- Используй простые структуры: списки, словари, множества, сортировку, префиксы, бинарный поиск.
- Давай переменным короткие понятные имена в контексте задачи: `left`, `right`, `mid`, `cnt`, `best`.
- Пиши линейно: сначала парсинг, затем вычисление, затем `print`.
- Для локальных проверок оставляй короткие принты только в черновике, перед отправкой коментрируй их.

## 4) Чего избегать (часто выглядит «не по-твоему»)

- Слишком «идеальной» архитектуры для простой задачи.
- Перегруза type hints (`dict[str, list[tuple[int, int]]]` в каждой строке).
- Избыточно длинных docstring и объяснений очевидного.
- Неестественно длинных имен везде (`current_optimal_candidate_value`).
- Функционального «шума» ради стиля: лишние обертки, фабрики, абстракции.

## 5) Практические правила на отправку

- Проверь формат ввода строго по условию (`n k` в одной строке или на разных).
- Убери отладочные `print`.
- Проверь крайние случаи: минимумы/максимумы, пустые/одинаковые значения, один тест.
- Не добавляй «защитный» код, если условие и так гарантирует корректность данных.
- Код должен читаться за 30 секунд: сразу видно, где ввод, где логика, где ответ.

## 6) Мини-чеклист перед сабмитом

- Есть `main()`.
- Нет лишних импортов.
- Нет случайных глобальных переменных без причины.
- Нет отладочного вывода.
- Пройдены примеры из условия.
